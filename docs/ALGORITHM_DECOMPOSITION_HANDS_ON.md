# ALGORITHM_DECOMPOSITION_HANDS_ON: 我在你旁边教你拆算法，从 0 到 1 接入

这是一篇“手把手”的文档。
它不是模板，而是你在旁边，我一步步带你拆算法的过程。
你会看到：
- 我会提醒你常见的错误
- 我会解释“为什么那样做会出问题”
- 我会给你具体拆解路径和操作步骤

它的密度会接近 `WORKFLOW_END_TO_END.md`。
你读完应该能做到：
**拿到任何算法 → 判断拆哪层 → 接入框架 → 验证有效**。

---

## 0. 先把“拆解的脑内地图”放好

拆解不是为了好看，而是为了控制变量。
你真正要做的不是“写出算法”，而是“知道自己改了什么”。

所以你需要一张脑内地图：

- Pipeline：解怎么生成、怎么修复、怎么保证硬约束
- Bias：你对解的倾向和偏好
- Adapter：搜索过程本身（提案/接受/更新）
- Plugin：统计、监控、并行、外部信号

只要你记住这四件事，你就不会迷路。

---

## 1. 你最容易犯的三类错误（先说清楚）

### 错误 1：把偏置写进算法

你会很自然地这样写：

- 在 SA 里写“安全惩罚”
- 在 VNS 里写“可行性奖励”

这样做的问题：
- 你以后换算法要抄一遍
- 你无法区分“算法变好”还是“偏好变了”

更好的做法：
- 把偏好变成 Bias
- 算法只负责搜索过程

### 错误 2：把修复写进算法

你会觉得“修复就是算法的一部分”，于是：
- 在算法循环里 repair
- 在 evaluate 里夹杂修复

问题是：
- 修复逻辑被绑死在算法里
- 每换一个算法就重写一次

更好的做法：
- 把修复写进 Pipeline
- 所有算法共用硬约束管线

### 错误 3：把统计写进算法

你会觉得“顺手 print 一下曲线、写个 csv 也没事”，
但这会让你：
- 并行时崩溃
- 算法被 I/O 污染
- 复用难度陡增

更好的做法：
- 统计、日志、可视化交给 Plugin

## 2. 拆解的第一步：先“看清算法在干什么”

当你拿到一个算法，先不要写代码。
先用一句话描述它的本质动作：

- SA：温度驱动的接受/拒绝
- A*：估价驱动的节点扩展
- Trust-Region：局部模型驱动的邻域搜索

把这句话写下来，你就能判断它是 Adapter 还是 Bias 或 Pipeline。

如果你不知道，把算法核心动作拆成 3 个动词：
- propose
- update
- accept

这三件事不变的，大概率是 Adapter。

## 3. 例子一：模拟退火 SA —— 典型的 Adapter 化

### 3.1 你可能会这样做（但不推荐）

很多人第一次写 SA，会在里面塞三件事：
- 修复逻辑
- 约束惩罚
- 日志输出

这样看似省事，但代价是：
- 你很难替换 SA
- 你无法对比 SA 与 VNS

### 3.2 正确拆解方式

SA 只关心“接受准则”，所以是 Adapter。

**拆法**：
- Adapter：SimulatedAnnealingAdapter
- Pipeline：负责可行性生成与修复
- Bias：可选（探索/收敛倾向）
- Plugin：可选（温度/接受率曲线）

### 3.3 实操步骤（接入流程）

1) 先写问题的 evaluate
2) 把修复逻辑放进 Pipeline
3) 装配 SA Adapter
4) 接 BenchmarkHarness 输出口径
5) 用 Run Inspector 检查组件是否齐全

### 3.4 验证方式

- 只换 Adapter（SA → VNS），其余不变，看结果是否可对比
- 关闭 Bias，看 SA 是否仍能跑

这样你就能明确“变好是算法还是偏好”。

## 4. 例子二：图论/路径规划（A* / Beam）

### 4.1 误区：把 A* 当万能

A* 在理论地图上很好用。
但现实路径规划的问题不是搜索，而是约束：
- 禁飞区
- 转弯半径
- 动态障碍
- 时间窗

A* 本身不会处理这些。

### 4.2 正确拆解

- Adapter：A* / Beam（负责扩展策略）
- Pipeline：约束生成 + 修复
- Bias：安全优先（可选）
- Plugin：节点扩展统计

### 4.3 实操步骤

1) 先把路径表示写进 Pipeline
2) 把约束修复放进 repair
3) A* 只负责节点扩展
4) 所有扩展出的候选必须走 Pipeline 统一修复

### 4.4 这样拆的收益

- A* 变成可复用的“策略内核”
- 约束逻辑只写一次
- 复杂路径规划终于可落地

## 5. 例子三：纯偏置化（真实可用的做法）

这里用框架里**已经存在**、且经常被用到的偏置模式举例：
**CallableBias + penalty/reward 规则**。

它的本质是：你不改搜索过程，只是把“偏好/惩罚”显式化。
这就是最标准的偏置化。

### 5.1 你可能会做的错事

- 把惩罚写进 evaluate
- 或把惩罚写进 SA / NSGA2

这样做的结果是：
- 你换算法就要重写惩罚
- 你没法单独验证惩罚是否有效

### 5.2 正确拆解（框架真实已有做法）

- Bias：CallableBias（mode="penalty" 或 "reward"）
- Adapter：保持原样
- Pipeline：保持原样

如果你需要范例，可以直接参考：
`docs/user_guide/bias_baby_guide.md` 里的 penalty 示例。

### 5.3 实操步骤

1) 先用原算法跑一遍（无惩罚）
2) 加一个 CallableBias（penalty）
3) 比较可行性比例、目标值是否变化

### 5.4 你会得到的收益

- 惩罚逻辑可复用
- 你能清楚看到“惩罚带来的变化”

## 6. 例子四：纯管线化（真实可用的做法）

这里用框架里**已经存在**的管线修复例子：
**ClipRepair / 生产调度的 Repair**。

很多“约束优化算法”其实没有改变搜索策略，
只是把解修回合法域。
这就是 Pipeline 的工作。

### 6.1 你可能会做的错事

- 在每次迭代里写“投影/裁剪”
- 把修复逻辑写在算法内部

这样做的问题是：
- 你每换一个算法，都要重写修复
- 你无法保证所有策略走同一套硬约束逻辑

### 6.2 正确拆解（框架真实已有做法）

- Pipeline.repair：ClipRepair（或生产调度中的 Repair）
- Pipeline.init：可行性初始化（如果有）
- Adapter：保持原样

例子可参考：
`examples/trust_region_dfo_demo.py`（ClipRepair）  
`examples/cases/production_scheduling/refactor_pipeline.py`（复杂修复）

### 6.3 实操步骤

1) 把修复函数写进 repair
2) 所有算法统一走 Pipeline
3) 记录 repair 触发率

### 6.4 验证

- repair 触发率是否下降？（说明 init/mutate 更友好）
- 修复后目标值是否更稳定？

## 7. 例子五：复杂 DFO（信赖域 + 代理模型 + 可信区间）

### 7.1 误区

把模型拟合、信赖域更新、采样策略全部写进一个大循环。
最终无法维护，也无法复用。

### 7.2 正确拆解

- Adapter：TrustRegionDFO / Subspace
- Plugin：MASModelPlugin（代理模型拟合）
- Bias：不确定性驱动偏置
- Pipeline：约束修复

### 7.3 实操步骤

1) 先用 TrustRegionDFO Adapter 跑通
2) 再挂 MASModelPlugin（增加代理模型）
3) 最后加 Bias（不确定性偏置）
4) 每加一步跑一次对比

### 7.4 验证

- 关闭 Plugin，算法是否还能跑
- Bias 打开后，探索/收敛是否变化
- 子空间学习是否提升收敛速度

---

## 8. 例子六：多策略协同（复杂拆解）

### 8.1 误区

把协同写成“并行跑多个算法”。
这样只能得到“堆叠”，无法解释因果。

### 8.2 正确拆解

- Adapter：MultiStrategyControllerAdapter
- Plugin：ParetoArchive / BenchmarkHarness
- Bias：各策略独立挂
- Pipeline：主管线统一

### 8.3 实操步骤

1) 先用两个策略协同（比如 SA + VNS）
2) 挂共享归档
3) 用 ModuleReport 看策略贡献

### 8.4 验证

- 各策略是否真的共享状态
- 是否有策略绕过主管线

## 9. 拆解后的“强制验证问题”

你每拆一个算法，都建议回答：

1) 它在组件层面是什么？
2) 解决了什么具体问题？
3) 固定其他组件能否单独验证？
4) 参数敏感度如何？
5) 哪些问题类型有效？哪些无效？
6) 与现有组件相比优势在哪？
7) 计算开销是否值得？

如果你能回答，说明拆解是有效的。

---

## 10. 当你真的不知道怎么拆时

记住四句话：

- 生成/修复解 → Pipeline
- 偏好倾向 → Bias
- 搜索过程 → Adapter
- 统计监控 → Plugin

这就是最稳的拆解原则。

---

## 11. 最后再说一句

你不是在写一个算法。
你是在设计一个“可复现、可对比、可进化”的实验系统。

拆解不是为了多写代码，而是为了让你能清楚地说：
**“我到底改了什么，所以结果才变了。”**
