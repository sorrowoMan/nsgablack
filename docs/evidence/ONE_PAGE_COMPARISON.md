# 一页对比：框架化方法 vs 传统写法

目的：用一页材料回答“这个框架到底减少了什么成本”。

## 对比对象

- A 组（传统写法）：问题、约束、偏好、策略、工程代码混在单脚本或少量脚本中。
- B 组（框架写法）：按 `Problem / Pipeline / Bias / Adapter / Plugin` 分层，入口为 `build_solver.py`。

## 成本对比（可直接放报告）

| 维度 | 传统写法（A） | 框架写法（B） | 差异结论 |
|---|---|---|---|
| 新增一个业务偏好 | 常改主循环，回归点多 | 主要改 `bias/`，装配层改动少 | B 更稳 |
| 替换搜索策略 | 常与问题逻辑耦合 | 主要改 `adapter/`，问题层不动 | B 更快 |
| 加并行/日志/回放 | 常侵入求解器主流程 | 主要加/换 `plugins/` | B 更低风险 |
| 复现实验 | 参数散落、难复盘 | `run_id + snapshot + modules/bias report` | B 更可审计 |
| 新人接手 | 隐式约定多 | 目录边界 + `doctor` + 注册说明 | B 更可维护 |

## 推荐量化指标（最少三项）

- 开发成本：完成一次“新增偏好 + 跑通”的净开发时长（小时）。
- 复现成本：新人按文档复现同结果所需时间（分钟）。
- 扩展成本：新增一个工程能力（如并行插件）改动文件数。

## 采样建议

- 用同一个问题做 A/B 两组，不换数据、不换目标。
- 每组至少 3 次记录，取中位数。
- 所有 run 保留到 `runs/`，并记录 `structure_hash`。

## 结论模板（可直接替换）

> 在同一问题与数据条件下，框架写法在“新增偏好、替换策略、工程能力接入”三项任务上均减少了改动面与回归风险；复现路径由“口头约定”变为“run_id + snapshot + 结构哈希”的可审计流程。
