# 一页对比：框架写法 vs 传统写法

目的：用一页材料回答“框架到底降低了什么成本、提高了什么可控性”。

## 对比对象

- A 组（传统写法）：问题、约束、偏好、策略、工程逻辑混在少量脚本中
- B 组（框架写法）：按 `Problem / Pipeline / Bias / Adapter / Plugin` 分层，入口为 `build_solver.py`

## 成本对比（可直接放报告）

| 维度 | 传统写法（A） | 框架写法（B） | 结论 |
|---|---|---|---|
| 新增业务偏好 | 常改主循环，回归面大 | 主要改 `bias/`，装配层小改 | B 更稳 |
| 替换搜索策略 | 常与问题逻辑耦合 | 主要改 `adapter/`，问题层不动 | B 更快 |
| 接入并行/日志/缓存 | 常侵入求解器内部 | 主要增删 `plugins/` | B 更低风险 |
| 实验复现 | 参数散落，难回放 | `run_id + snapshot + module report` | B 更可审计 |
| 新人接手 | 隐式约定多 | 有目录边界 + doctor + catalog | B 更可维护 |

## 推荐量化指标（至少 3 项）

- 开发成本：完成一次“新增偏好 + 跑通”的净时长（小时）
- 复现成本：新成员按文档复现实验的时长（分钟）
- 扩展成本：接入一个工程功能（如并行插件）涉及改动文件数

## 采样建议

- 同一问题、同一数据做 A/B 对比
- 每组至少 3 次，取中位数
- 所有 run 保留 `structure_hash`，确保“结构可比”

## 结论模板（可直接替换）

在同一问题与数据条件下，框架写法在“新增偏好、替换策略、接入工程能力”三类任务上显著降低改动面与回归风险；实验复现从“口头约定”转为“run_id + snapshot + 结构哈希”的可审计流程。
