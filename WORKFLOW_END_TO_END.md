# WORKFLOW_END_TO_END: 我在你旁边，从 0 到 1 跑通一个优化问题（可复现/可对比）

你手上有一个“真实问题”，但它还不是一个“能跑、能复现、能对比”的实验。
这份文档不是讲概念，而是“手把手陪你跑通一次闭环”。

如果你现在的状态是这样的，没关系——这正是正常起点：

- 你已经能写 `evaluate(x)`，但每次跑出来结果差异很大
- 你想加并行/日志/统计，但不想把算法代码弄脏
- 你脑子里有一堆算法想法（NSGA2、VNS、SA、多策略协同），但不知道先从哪里下手  
  这里的“多策略协同”指的是**多策略提案者 + 统一评估/归档/调度**的编排，而不是“多智能体”概念的硬套。

我们这趟旅程的目标很明确：**把最小闭环跑通，之后你才能稳定迭代**。

你会发现我在强调“先定工程口径”。这不是形式主义：现实里最常见的失败不是算法不好，而是你根本不知道“到底哪里变好了 / 变坏了”。 

你最终会得到：

- 一段可以反复跑的脚本（固定 seed 可复现，多 seed 可稳健）
- 一套统一输出（`progress.csv` + `summary.json`），方便对比
- 一组可组合的组件（你只改一个点，就知道影响来自哪里）

**传统做法 vs 框架做法（先给你一个对照感）**

- 传统做法：把问题、算子、统计、输出都塞进同一个算法脚本里，改一次就牵一片  
- 框架做法：把“问题/表示/偏置/策略/能力”拆开，保证每次改动只影响一个点  
- 为什么更好：你能明确说“我改了什么 → 结果为什么变”，不靠猜、不靠记忆

**设计哲学：单一职责带来的“自然感”**

- 每个组件只关心一件事：输入是什么、怎么处理、输出什么  
- 你不需要思考“这段逻辑到底放哪”，因为职责边界已经被固定  
- 这让开发过程从“多维纠结”变成“线性推进”：先想清楚组件契约，再实现  
- 结果是：组件开发轻松、组合自然、问题定位极快

## 先建立"脑内地图"：5 个核心组件 + 2 个组织方式

在 NSGABlack 里，核心闭环由 5 个组件组成：

- `Solver`：运行容器与生命周期（负责调用 adapter/pipeline/bias/plugins）
- `RepresentationPipeline`：初始化/变异/修复/编码解码（硬约束优先）
- `BiasModule`：软约束/偏好/算法倾向（可组合、可开关）
- `Adapter`：策略内核（怎么提案/怎么消费反馈/怎么推进搜索过程）
- `Plugin`：能力层（并行、记录、监控、短路评估等）

> 说明：如果你用 `BlackBoxSolverNSGAII`（搜索逻辑内建），不需要显式配 Adapter；如果你用 `ComposableSolver` / `BlankSolverBase`，Adapter 就是决定"怎么搜索"的核心引擎。

组织方式（帮你不漏配、好复用、好发现）：

- `Suite`：权威组合（把必配组件一次装配好，避免漏配）
- `Catalog`：可发现性（search/show/list）

### 默认必配组合清单（避免你“漏线”）

下面这些是“经常被忘记但必须成套”的组合，优先用 Suite 收口：

- **信号驱动偏置** → 必须配信号插件（如 `MonteCarloEvaluationPlugin`），否则偏置会“悄悄失效”推荐：`suite.monte_carlo_robustness`
- **动态优化** → 必须配动态信号提供方（如 `CLISignalProvider` 或自定义 SignalProvider）推荐：`suite.dynamic_switch`
- **多策略协同** → 必须配共享信息与归档（不然协同退化为并行堆叠）推荐：`suite.multi_strategy`（默认带 ParetoArchive）  
  本质是 Strategy/Adapter 的编排框架：多策略提出解，统一评估、统一归档、统一调度。
- **可复现实验** → 必须配统一输出口径（否则无法对比/复现）
  推荐：`suite.benchmark_harness`

你可能会想：

> “我直接写一个大算法脚本不是更快吗？”

是的，短期更快。但你一旦要改一个点，或者对比两个结果，就会变成“记忆力实验”。这套解耦就是为“长期迭代”设计的：

- 新约束/新偏好 -> 往 `BiasModule` 里加（不动策略）
- 表示/算子/修复 -> 放进 `RepresentationPipeline`（不动问题、不动策略）
- 搜索策略 -> 用 `Adapter` 替换/串联（不动问题、不动表示）
- 并行/记录/短路/监控 -> 用 `Plugin` 上能力（不动底座）
- 容易漏配的“伙伴组件” -> 收敛成 `Suite`（减少低级错误）

## 运行前审查（Run Inspector）：避免“忘记挂”

你会很快遇到这个问题：

> “我好像装了某个插件/偏置/管线，但结果像没生效？”

这时建议在运行前做一次可视化审查（Run Inspector）：

- 展示当前 solver wiring（底座/管线/偏置/插件/多策略）
- 支持一键勾选/取消（即开即关，偏置/管线/插件都可即时开关）
- 基于 Catalog 的 `companions` 提示缺失伙伴（高亮提醒，不自动装配）
- 若启用 ModuleReport，会把 `ui_snapshot` 写进报告，方便复盘“运行前到底勾了什么”

命令：

```bash
python utils/viz/visualizer_tk.py --entry path/to/your_script.py:build_solver
```

统一入口：

```bash
python -m nsgablack run_inspector --entry path/to/your_script.py:build_solver
```

## 0) 先把“实验口径”定死（非常重要）

你第一反应大概率是：

> “我能不能先跑起来，后面再补工程口径？”

可以，但你很快会遇到第二个问题：

> “我改了个参数/换了个算子，结果忽好忽坏，到底是变强了还是随机数在作怪？”

这就是口径没定死导致的。解决方法很朴素：**把口径写死，让每次运行都能留下同一种证据**。

> 传统做法：每个算法脚本自己写 CSV / 打 log / 输出结果，最后对比口径不一致  
> 框架做法：统一交给 BenchmarkHarness + ModuleReport，输出格式固定  
> 为什么更好：你能确保“同口径对比”，而不是“结果看起来差不多”

### 你要先决定这三件事：

- **输入是什么**：seed、预算（steps/evals/time）、输出目录
- **输出是什么**：`progress.csv`（过程曲线）、`summary.json`（终局摘要）
- **谁负责写**：用 Plugin 写，不要塞进算法循环

### 关于 seed 的正确用法

你可能会问：

> “到底固定 seed 还是多次运行？”

答案是“两种都要”：

- 复现/对比模式：固定 seed（你能明确对比“我改了什么”）
- 鲁棒/稳健模式：多 seed 多次跑（用统计指标做决策）

无论哪种，都要把实际 seed 写入 `summary.json`，否则过几天你追不回结果。

### 推荐做法（能力层，不污染底座）

- 用 `BenchmarkHarnessPlugin` 负责输出口径
- 用 `suite.benchmark_harness` 一次性装配（避免忘记接线）

当你做到这一步，你就拥有了“可比较”的基础设施。

## 1) 定义问题（Problem）：先把“你到底在优化什么”讲清楚

这一步你只做一件事：写一个稳定的 `problem.evaluate(x)`。

你可能会想：

> “我顺手把日志/画图/统计塞进去不就完了？”

短期看起来很爽，长期会反噬（尤其并行时）：

- `evaluate()` 里写文件/画图会相互抢资源，轻则慢，重则卡死
- 换算法/换入口脚本时，问题定义混入大量“副作用逻辑”

更稳妥的分工是：

- `evaluate()` 只做“把 x 变成 objectives/constraints”
- 记录/可视化/统计 = Plugin 的工作（可开关、可替换）

> 传统做法：evaluate 里写文件/画图/统计，跑并行就炸  
> 框架做法：evaluate 只评估，统计交给 Plugin  
> 为什么更好：评估函数变成“纯函数”，并行稳定、可复现

提醒：**不要在 `evaluate()` 里写文件/画图/发请求**。并行评估时会互相抢资源。

输入

- 决策变量：维度/变量名/上下界（bounds）
- 目标：单目标或多目标
- 约束（可选）：建议用 `g(x) <= 0` 表示“可行”

输出（契约）

- `problem.evaluate(x) -> objectives`
- `problem.bounds`（支持 list 或 dict）

补充一句：`evaluate_constraints` 是可选的，**硬约束优先放到 pipeline 里做（repair/init）**。

## 2) 组装表征管线（RepresentationPipeline）：先把“可行解怎么来”解决掉

你很快会撞的第一堵墙是：

> “算法还没发挥，解先不合法了。”

所以先把“产生候选解的机器”搭好：

- 能初始化一个还不错的起点（最好是可行解）
- 能在邻域里变异（尽量不把自己变成垃圾解）

你可能会说：

> “可行性我写进算法里处理不就行了？”

可行，但很亏：你一换算法，就要重新搬一遍可行化/修复逻辑。

更稳的做法：把硬约束前置到 pipeline（init/mutate/repair）。好处是：

- 换算法时不用重复实现可行化
- 算法更专注于“怎么搜索”，而不是“怎么修”

> 传统做法：每个算法里手写可行化逻辑，换算法就重复  
> 框架做法：硬约束放到 pipeline，算法共享  
> 为什么更好：约束逻辑不重复，错误更少，维护更轻

**别忘记接 `repair`**。没有 repair，你会在非法解里打转，看起来能跑，实际上在原地踏步。

一个典型“漏配坑”

- VNS 这类邻域搜索常常需要 context/邻域切换 mutator；忘了接会退化成“在同一邻域打转”。
- 推荐：优先用 `suite.vns`，别手搓到漏线。

快速自检（很值）

- init 出来的 x 是否多数合法（至少 90%）
- mutate 后是否总被 repair 拉回（总拉回说明 mutator 太“野”）

## 3) 组装偏置（BiasModule）：把“想要的倾向”外置

如果 pipeline 管“硬约束”，BiasModule 就是“你脑子里的偏好”。

你经常会说：

- “我更想要可行解，宁愿目标差一点”
- “我讨厌某种结构/模式（哪怕分数好看）”
- “这个空间太大，我希望搜索更激进/更保守”

你当然可以把这些写进算法，但会带来两个后果：

- 规则一变化，你就要改算法（很快变成 N 个版本）
- 你很难确认“这次变好，是偏好变了还是策略变了”

BiasModule 的意义：把倾向外置、可开关、可组合、可复用。

> 传统做法：把偏好/惩罚写进算法或 evaluate，改一次就要改算法  
> 框架做法：偏好封装为 Bias，可开关、可组合  
> 为什么更好：你能清楚分辨“策略变了”还是“偏好变了”

提醒一个高频坑：

**signal-driven bias 必须挂对应的评估/统计插件**（如 `MonteCarloEvaluationPlugin`），否则偏置拿不到信号，会“悄悄失效”。
建议同时挂 `ModuleReportPlugin`，能审计“偏置启用了但信号未注入”。

## 4) 选择策略内核（Adapter）：你的“搜索过程”在这里

Adapter 只回答一个问题：

> “下一步我往哪里搜？”

你常见的冲动有两种：

1) 选一个现成算法直接跑（最稳）
2) 一上来就想做融合（最常见）

融合可行，但建议从“阶段式串联”开始（先探索后精修）。好处：

- 过程可解释
- 可做消融（删掉某段看影响）

经验法则：**先串联验证，再考虑深度融合**。

> 传统做法：把多个算法硬混在一个循环里，过程不可解释  
> 框架做法：Adapter 负责策略内核，组合走多策略协同或阶段式串联  
> 为什么更好：每个策略的贡献可拆解、可消融、可对比

### 多策略协同（你会越来越常用）

**先澄清概念**：这里不是“多智能体”叙事，而是**多策略提案者 + 统一评估/归档/调度**。  
所以它更像是一个 Strategy/Adapter 的编排框架，而不是一个独立“智能体算法”。

协同不是“多算法堆在一起”，而是**有调度、有共享状态、有统一口径**。

协同的核心要素：

- 共享状态：例如 `ParetoArchive`、当前最好解、统计信号
- 资源预算：每个策略的评估预算/时间配额
- 统一评估口径：所有策略产出的解必须过同一条主管线（repair + evaluate）

常见坑：

- 资源倾斜：某策略吞掉预算，但质量差
- 反馈延迟：策略看不到最新共享信号
- 口径不统一：某策略绕过主管线，结果不可比

推荐最小协同装配：

- `suite.multi_strategy`
- `ParetoArchivePlugin`
- `BenchmarkHarnessPlugin`
- `ModuleReportPlugin`

一句落地法则：

> 先把协同当“调度工程”，不是“算法融合”。

## 5) 加能力层（Plugins）：把“工程能力”外置

你可以把 print/写 CSV/并行全塞进算法循环里，但会带来三连副作用：

- 算法逻辑被 I/O 淹没
- 换算法就要复制一份输出逻辑
- 一上并行就容易输出乱/卡死

所以能力要外置：Plugin 负责能力，算法负责决策。

> 传统做法：并行/日志/统计塞进算法循环，越改越乱  
> 框架做法：工程能力交给 Plugin，算法保持干净  
> 为什么更好：算法可复用，工程能力可替换

并行评估提醒：多进程后端要求 `problem` 可序列化，否则要用 `problem_factory`。

## 6) 用 Catalog 找组件，用 Suite 一键装配（你会越来越依赖它）

当你开始组合多策略、多偏置、多管线时，你的记忆一定不够用。
Suite 的价值不是“你不会装配”，而是**你会漏配**。

> 传统做法：靠记忆装配，漏了也不报错，结果默默退化  
> 框架做法：Suite 收口必配组合，Catalog 提醒伙伴依赖  
> 为什么更好：减少“隐性退化”，装配可追溯

建议先用 Suite 跑通，再替换其中某个组件做实验。

- 搜索：`python -m nsgablack catalog search vns`
- 查看：`python -m nsgablack catalog show suite.vns`

如果你遇到 `No module named nsgablack`，通常是路径问题：

```powershell
python -m pip install -e .
```

## 7) 运行一次可复现实验（把闭环跑通）

先做一次小预算 smoke run：

- 预算很小（如 `max_steps=20`）
- 目标是“闭环跑通”，不是“结果最好”

检查两件事：

- `progress.csv` 是否持续写入
- `summary.json` 是否包含 seed/budget/wall_time/eval_count

如果这些都对，你已经赢了一半。

参考示例：

- `examples/end_to_end_workflow_demo.py`
- `examples/notebooks/workflow_end_to_end_portfolio.ipynb`

## 8) 迭代与对比（这就是框架最擅长的事）

一次只改一件事，再复跑。

你会很快发现：框架不是替你想算法，而是替你“控制变量”，让你的算法想法能被验证、能被复用。

最后给你一句陪跑式收尾：

当你哪天觉得“这个问题终于被我摸透了”，你回头看会发现，你做的不是某一个算法，而是一整套可复用的工程路径——问题接口、表示、偏置、策略、能力、口径，全都能带走。
