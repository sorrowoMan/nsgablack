# WORKFLOW_END_TO_END: 我就在你旁边，带你从 0 到 1 落地一个优化问题（可复现/可比较）

你现在手上有一个“真实问题”，但它还不是一个“能跑、能复现、能对比”的实验。

我猜你现在的状态大概率是这样的：
- 你已经能把 `evaluate(x)` 写出来了，但每次跑出来的结果都不太一样
- 你想加点“工程能力”（并行/日志/统计），又担心把算法代码弄脏
- 你脑子里有很多算法想法（NSGA2、TS、VNS、SA、并行协同），但不知道应该先从哪里下手

这份文档的目标很简单：我们一起把它落成一个最小闭环，然后你就可以开始稳定迭代了。

你会发现我一直在“强迫你先做工程口径”。这不是形式主义：真实项目里，最常见的失败不是算法不好，而是你根本没法确认“到底哪里变好了/变坏了”。

你最终会得到：
- 一段可以反复跑的脚本（你可以固定 seed 做复现，也可以多 seed 多次运行做更稳的统计）
- 一套统一输出（`progress.csv` + `summary.json`），方便你和自己/同学/论文对比
- 一组可组合的组件：你只改一个点，就能知道改动带来的效果

在 NSGABlack 里，最小闭环只需要 4 个一级公民：
- `Solver`：运行容器与生命周期（负责调用 adapter/pipeline/bias/plugins）
- `RepresentationPipeline`：初始化/变异/修复/编码解码（硬约束优先）
- `BiasModule`：软约束/偏好/算法倾向（可组合、可开关）
- `Plugin`：能力层（并行、记录、监控、短路评估等）

其它概念只是“组织方式”（帮你不漏配、好复用、好发现）：
- `Adapter`：策略内核（怎么提案/怎么消费反馈/怎么推进搜索过程）
- `Suite`：权威组合（把必配组件一次装配好，避免漏配）
- `Catalog`：可发现性（search/show/list）

你可能会想：为什么要把东西拆得这么开？我直接写一个“大算法脚本”不是更快吗？

我猜你一定经历过这些时刻（甚至现在就在经历）：
- 你突然有一个新点子：想把 `NSGA2 -> VNS` 串一下，或者加一个“角色协同”的小实验，最好 30 分钟就能跑出结果
- 业务方临时加了新约束/新偏好：你不想去改算法主循环，怕改崩、怕引入隐性 bug
- 性能出了问题：评估太慢/并行不稳定/日志把进程卡死，你只想换一套并行评估或记录方式，不想重写算法
- 你要做对比：你希望“只改一件事”，否则你永远不知道提升来自哪里

这套解耦就是为这些场景服务的：
- 新约束/新偏好 -> 往 `BiasModule` 里加（不动策略）
- 表示/算子/修复 -> 放进 `RepresentationPipeline`（不动问题、不动策略）
- 搜索策略 -> 用 `Adapter` 替换/串联（不动问题、不动表示）
- 并行/记录/短路/监控 -> 用 `Plugin` 上能力（不动底座）
- 容易漏配的“伙伴组件” -> 收敛成 `Suite`（减少低级错误）

## 0) 我们先把“实验口径”定死（非常重要）

先别急着选算法。你只要后面想“比较/复现/迭代”，这一步就必须先做。

你现在要做的是：决定这次实验“输入是什么”“输出是什么”。

这时候你可能会问：

“我先把算法跑起来不行吗？为什么还要管这些工程细节？”

当然可以先跑起来，而且你一定会这么做（我也会）。但你很快会遇到第二个问题：

“我改了个参数/换了个算子，结果忽好忽坏，我到底是变强了，还是只是随机数变了？”

如果你没有统一口径，这个问题没法回答。你会开始靠感觉调参、靠截图对比、靠‘这次看起来更顺眼’做决定。

解决方法也很朴素：把实验口径写死，让每次运行都能留下同一种可比对的证据。

这里最容易产生分歧的是 `seed`。

这时候你会问：

“我到底该固定 seed，还是应该多跑几次、用更多随机性？”

这完全正确。在工程场景里，`seed` 更像是“实验编号/随机源”，它的正确用法通常是两种模式并存：
- 复现/对比模式：固定 `seed`，你才能明确对比“我改了什么 -> 结果怎么变”
- 鲁棒/搜更广模式：不固定 seed（或给一个 seed 列表），跑多次后用统计指标做决策

不固定 seed 可行吗？可行，而且常常更实用。但有一个底线：把“这次实际用的 seed”记录进输出（例如 `summary.json`），否则你过几天根本追不回这次结果是怎么来的。

输入（建议写进 config 或脚本参数）
- `seed`: int | None，随机种子（可选；不传则走随机，但仍建议记录到输出）
- `budget`: dict，预算（至少选一种）
  - `max_steps`: int
  - `max_evaluations`: int
  - `wall_time_s`: float
- `output_dir`: str，输出目录（建议自动带 `run_id`）

输出（建议统一写出）
- `progress.csv`: 逐步曲线（step/eval_count/best_score/...）
- `summary.json`: 终局摘要（seed/time/eval_count/best_score/...）

推荐做法（能力层，不污染底座）
- 用 `BenchmarkHarnessPlugin` 负责输出口径
- 用 `suite.benchmark_harness` 一次性装配（避免你忘记接线）

当你做到这一步，你就拥有了“可比较”的基础设施。

你可以自检一下：现在你应该能很自然地回答这三个问题——
- “这次实验用的 seed 是多少？”
- “我用的预算口径是什么（step/eval/time）？”
- “我把过程曲线和终局摘要写到了哪里？”

如果你读到这里有点“反直觉”（怎么还没开始选算法），那说明你正在走一条更稳的路：先把地基钉牢。接下来我们再把“问题本身”封装清楚。

## 1) 定义问题（Problem）：先把“你到底在优化什么”讲清楚

先别急着“上算法”。这一步你只做一件事：把问题写成一个足够稳定的 `problem.evaluate(x)`。

你会本能地想：既然我都写 evaluate 了，我顺手把日志/画图/统计也塞进去不就完了？
这么做当然“能跑”，而且短期还挺爽。

但你很快会碰到反噬（尤其是一旦并行起来）：
- 并行评估时，`evaluate()` 里的写文件/画图会互相抢资源，轻则慢，重则输出乱/卡死
- 你换算法/换入口脚本时，会发现问题定义里混进了大量“与问题无关”的副作用

更稳妥的分工是：
- `evaluate()` 只负责“把 x 变成 objectives/constraints”
- 记录/可视化/统计 = Plugin 的工作（你想开就开，想关就关）

那是不是 evaluate 里完全不能做任何额外事情？
不是。你可以做一些“不会影响结果的纯记录”，但建议满足两个条件：可开关 + 不依赖全局状态（否则迟早被并行搞炸）。

输入
- 决策变量：维度/变量名/上下界（bounds）
- 目标：单目标或多目标
- 约束（可选）：建议用 `g(x) <= 0` 表示“可行”

输出（契约）
- `problem.evaluate(x) -> objectives`
  - 单目标：shape `(1,)` 或标量
  - 多目标：shape `(M,)`
- `problem.bounds`
  - 支持 `[(low, high), ...]` 或 `{name: (low, high)}`

工程建议（你会少踩很多坑）
- `evaluate()` 只做评估，不在里面写文件/画图（这些交给 Plugin）

## 2) 选择/组装表征管线（RepresentationPipeline）：先把“可行解怎么来”解决掉

接下来你会撞的第一堵墙通常是：算法还没来得及发挥，你的解就先“不合法/不可行/全撞边界”。

所以这一段我们不谈策略，只把“产生候选解的机器”搭好：
- 能初始化一个还不错的起点（最好是可行解）
- 能在邻域里变异（并且尽量不把自己变成垃圾解）

你可能会说：可行性我写进算法里处理不就行了？
可行，而且很多传统实现就是这么写的。

但在框架里，这样做很亏：你一换算法，就要重新搬一遍“可行化/修复/边界处理”的同一套代码，最后算法文件里全是 if/else。

更适合长期迭代的做法是：把硬约束尽量前置到 pipeline（init/mutate/repair）。
你会立刻得到两个好处：
- 换算法时，不用重复实现“可行化”
- 算法更专注于“怎么搜索”，而不是“怎么把解修回去”

输入
- `initializer`: 生成可行解的策略
- `mutator`: 邻域变异算子（连续/整数/排列/图/矩阵等）
- `repair`（可选）: 修复（越界/不可行）
- `codec`（可选）: 编码/解码（当内部表示 != 评估表示）

输出（契约）
- `pipeline.init(context) -> x`
- `pipeline.mutate(x, context) -> x_new`
- `pipeline.repair(x, context) -> x_fixed`（如有）

一个特别典型的“漏配坑”
- VNS 这类邻域搜索通常需要一个“上下文切换/邻域切换”的 mutator；你如果忘了加，VNS 会退化成反复在同一邻域里打转。
- 所以：优先用 `suite.vns` 这类权威组合，别手搓到漏线。

你可以做一个很快的自检（这一步很值）：
- init 出来的 x 是否大多数都合法（至少 90% 以上）
- mutate 之后的 x 是否经常被 repair 拉回（如果每次都 repair，说明 mutator 太“野”了）

到这里，你基本上已经能稳定地产生“可用的候选解”了。下一步我们才开始把“你想要的倾向”加进去：有些倾向是业务的，有些倾向是算法的。

## 3) 组装偏置（BiasModule）：把你“想要的倾向”写成组件，而不是散在算法里

如果你把 pipeline 当成“硬约束与表示”，那 BiasModule 就是“你脑子里的偏好”。

现实里你经常会说这些话：
- “我更想要可行解，宁愿目标差一点”
- “我更讨厌某种结构/某种模式（哪怕它暂时分数好看）”
- “这个问题空间太大，我想让搜索更激进/更保守一点”

你当然可以把这些倾向直接写进算法里：加惩罚项、加规则、加一堆 if/else。
这么做可行吗？可行，甚至一开始最快。

不妥在哪里？当规则开始变化（几乎必然会变）：
- 你会不停地改算法、复制算法，最后变成“同名算法 N 个版本”
- 你很难清楚地说：这次结果变了，是因为“策略变了”，还是因为“偏好变了”

BiasModule 的意义就是把这些倾向外置：可开关、可组合、可替换，并且能跨算法复用。
你能明确地说：“我这次只换偏置，不换算法”，这对真实迭代非常重要。

输入
- 领域偏置（domain）：软约束/偏好（惩罚项、可行性优先级、业务规则）
- 算法偏置（algorithmic）：搜索倾向（退火温度、pattern search、tabu 记忆等）
- 信号驱动偏置（signal-driven）：依赖统计信号/metrics 的偏置
  - 这类偏置必须声明 `requires_metrics`，并强烈建议提供对应 `suite`（避免“只有偏置单独拿出来没用”）

输出（契约）
- `bias_module.apply(x, objectives, context) -> (objectives', metadata)`
  - 或者以“打分/排序倾向”的形式影响选择

## 4) 选择策略内核（Adapter）：你要的“搜索过程”在这里

现在才轮到“算法过程”登场。

你可以把 Adapter 想象成：它只回答一个问题——“下一步我往哪里搜？”

你可能会有两种冲动：
1) 选一个现成算法直接跑（最稳）
2) 一上来就想做融合（最常见）

融合可行吗？可行，但建议别从“完全融合”开始。

更稳的起手式通常是阶段式组合：先让一个策略负责“铺开搜索”（例如 NSGA2），再让另一个策略负责“深挖局部”（例如 VNS/SA）。
好处是：
- 过程可解释：每一段在干什么你很清楚
- 可做消融：删掉某一段看影响，不会全混在一起

等你跑出证据后，再把值得通用的部分下沉成 Bias/Representation/Plugin（让它更可复用）。

输入
- `pipeline`: 负责产生候选解
- `bias_module`: 负责倾向/软约束
- `context`: 提供运行时信息（seed/budget/统计信号/共享状态等）

输出（契约）
- `adapter.propose(context) -> candidates`
- `adapter.update(feedback, context) -> None`

你会很常见地遇到“我要融合算法”的想法：
- 经验法则：先按阶段串起来最稳，比如 `NSGA2 -> VNS`，别急着把 VNS 硬塞进 GA 的每一步。
- 你可以用 Suite/Controller 做阶段调度，把流程写清楚，后面也更好复现和对比。

如果你此刻已经能说清楚“我这段时间在做什么”（铺开/深挖/约束/并行），那你就已经在写一个可维护的系统，而不是在堆一个一次性的脚本。

## 5) 加能力层（Plugins）：你想要的“工程能力”，不要写进算法里

如果前面 1-4 是“把问题和算法跑起来”，那 Plugins 就是“给这台机器加外挂”。

你当然可以把 print/写 CSV/开进程全塞进算法循环里。
这条路也确实很多人这么走。

但它会带来非常典型的三连副作用：
- 算法逻辑被 I/O 逻辑淹没，你越改越怕
- 换算法/换入口脚本就要复制一份输出逻辑（复制久了必然分叉）
- 一上并行，日志/文件输出就容易混乱（Windows 多进程尤其明显）

所以框架里把这些能力抽成 Plugin：算法负责“决策”，Plugin 负责“能力”。
你想开就开，想关就关，不会污染底座。

常见插件能力
- 并行评估（ParallelEvaluator/parallel suites）
- 统一实验口径（BenchmarkHarnessPlugin）
- 记录/监控/短路评估

Plugin 输入/输出是什么
- 输入：通过 context keys 读取信息（例如 eval_count、wall_time、best_score 等）
- 输出：写文件/打点/统计/告警/把 signal 写回 context（供 signal-driven bias 使用）

## 6) 用 Catalog 找组件，用 Suite 一键装配（你会越来越依赖它）

你现在要做的是：别猜“VNS 在哪”“鲁棒性怎么接”，直接搜。

这时候你会问：

“我明明会组装，为什么还要 suite？”

你会组装不代表你不会漏配。真实开发里最阴间的 bug 往往是：
“能跑，但退化；看起来没报错，但其实少了一根线。”

少接一个 mutator、少挂一个统计插件、少写一个 context key，你得到的不是 crash，而是“默默变差”，然后你会开始怀疑人生。

Suite 的价值就是把这些“必配伙伴”收敛成一个权威入口：你先用权威组合跑通，再逐步替换其中某一块做实验。

陪跑式的经验：当你开始做“多策略/多偏置/多表示”的组合时，你的记忆一定不够用。Catalog/Suite 不是为了炫技，而是为了让你在复杂度上升时还能保持清醒。

- 搜索：`python -m nsgablack catalog search vns`
- 查看：`python -m nsgablack catalog show suite.vns`

如果你遇到报错：`No module named nsgablack`

这通常不是框架坏了，而是“你当前所在目录”导致 Python 找不到包：
- 当你 `cd` 进了仓库根目录（也就是包目录本身）时，`python -m nsgablack ...` 需要“父目录”在 `PYTHONPATH` 才能找到它

解决方法（选一个你喜欢的）

1) 最推荐：安装为可编辑包（之后在哪运行都行）

```powershell
python -m pip install -e .
python -m nsgablack catalog search context
```

2) 不安装、快速试用：把父目录加到 `PYTHONPATH`

```powershell
$env:PYTHONPATH=".."
python -m nsgablack catalog search context
```

3) 直接在父目录运行

```powershell
cd ..
python -m nsgablack catalog search context
```

原则
- 你当然可以手工组装，但推荐先用 Suite（避免漏配“伙伴组件”）
- Suite 是“官方推荐组合”，也是你未来维护成本最低的入口

如果你现在脑子里已经出现了“我想试试 A 组合、再试试 B 组合”的冲动，那恭喜：你已经进入了框架最舒服的使用状态。接下来你需要的不是更多抽象，而是把第一次实验跑通，让它开始产出可比对的数据。

## 7) 运行一次可复现的实验（把闭环跑通）

你现在要做的是：写一个最短脚本，先跑通“生成解 -> 评估 -> 记录输出”。

这时候你通常会想：

“我能不能直接把预算拉满、开很多并行、把所有偏置都挂上，然后看看能不能出奇迹？”

可以，但这往往是最快把自己搞崩的一条路。更稳的方式是先做一次小预算 smoke run：
- 预算很小（例如 `max_steps=20`），目标是“闭环跑通”而不是“结果最好”
- 输出落盘正确（CSV/JSON 都有，字段完整）
- 你能复现（同 seed 重跑结果一致，至少统计口径一致）

这个时候你会非常想看“它到底有没有在进步”。别急，先看两件事：
- `progress.csv` 有没有在持续写入（step/eval_count 是否递增）
- `summary.json` 里 seed/budget/wall_time/eval_count 是否都齐

如果这两件事都对，你已经赢了一半：你拥有了一个能持续迭代的骨架。

如果你想直接看一个可跑的版本：
- `examples/end_to_end_workflow_demo.py`

如果你更喜欢“直接可运行、边看边改”的形式：
- Notebook 版（一个更像真实任务的例子：组合约束 + 多目标 + Suite + 统一输出口径）：`examples/notebooks/workflow_end_to_end_portfolio.ipynb`

你自己的脚本通常可以很短：

```python
from nsgablack.core import ComposableSolver
from nsgablack.utils.suites.benchmark_harness import attach_benchmark_harness
solver = ComposableSolver(...)
attach_benchmark_harness(
    solver,
    output_dir="runs/demo",
    seed=42,
    budget={"max_steps": 200},
)
result = solver.solve()
print(result)
```
期望输出
- `runs/demo/.../progress.csv`
- `runs/demo/.../summary.json`
到这里为止，你已经不再“凭感觉做算法”了，你有了稳定的工程闭环。

你可以停下来喝口水：从此以后，你的每一次改动都能留下证据。无论是换算子、加偏置、加并行，还是把流程串成多阶段，你都有一套稳定的对照体系。

如果这一步没跑通，最常见的排查顺序是：
- `problem.evaluate()` 是否纯函数（有没有写文件/随机数没控 seed）
- pipeline 的 init/mutate 是否能返回合法 x（尤其是 shape/bounds）
- suite/plugin 是否正确挂载（输出目录/权限/并行参数）

## 8) 迭代与对比（这就是框架最擅长的事）

你现在要做的是：一次只改一件事，然后复跑，把“现实的不确定”变成“可控的改动”。

这时候你会问：

“我明明有很多想法，为什么要一次只改一件？”

因为真实世界里，最容易自欺欺人的是“我觉得它更好”。当你一次改三件事，你根本不知道是哪件在起作用；最后你只能靠叙事包装自己，而不是靠证据。

相反，一次只改一件事，你会得到一种很强的掌控感：你知道每一次改动的因果方向，久而久之你会形成自己的“问题-策略库”。

更细腻一点的做法（你会越用越顺手）：
- 先写一句“我这次要验证什么假设”（例如：`ContextSwitchMutator` 是否真的能提升 VNS 的探索）
- 只改动与该假设相关的 1 个组件
- 用同一口径跑 3~5 次（多 seed 或固定 seed 看你的目的）
- 只看你关心的 1~2 个指标（别被一堆图表带跑偏）

只改一件事，然后复跑：
- 换 mutator（不动 problem）
- 加一个 domain bias（不动 adapter）
- 把 `NSGA2 -> VNS` 变成 `NSGA2 -> SA`（不动 pipeline）
- 加并行评估 suite（不动算法逻辑）

核心目标：把“现实问题的信息不明确”变成“可持续迭代的配置空间”。

你会慢慢发现：框架不是替你想算法，而是替你“控制变量”，让你的算法想法更容易被验证、更容易被复用。

最后给你一句陪跑式的收尾：当你哪天觉得“这个问题终于被我摸透了”，你回头看会发现，你做的不是某一个算法，而是一整套可复用的工程路径——问题接口、表示、偏置、策略、能力、口径，全都能带走。
